| Назначение | Переменная в скрипте | Текущий канал | Что проверить на новом объекте |
|---|---|---|---|

# 2. Назначение файла

`507_Pressure_makeup.js` управляет клапаном подпитки системы отопления.

Основная задача:

```text
если давление отопления ниже рабочего порога
→ кратко открыть клапан подпитки
→ закрыть клапан
→ подождать стабилизацию давления
→ проверить результат
→ повторить до целевого давления или до maxPulses
```

Скрипт не должен держать клапан открытым постоянно. Работа только импульсная.

# 3. Принцип работы

## 3.1. Старт

При запуске скрипта через `setTimeout(..., 3000)` выполняется:

1. Логирование старта.
2. Принудительное закрытие клапана `CH.valveMakeup`.
3. Запись события: `Принудительное закрытие клапана на старте`.
4. Первичный вызов `evaluate()`.

Это важно: если до перезапуска `wb-rules` клапан был открыт, после старта скрипта он должен быть закрыт.

## 3.2. Проверка датчика давления

Скрипт читает два значения:

```text
CH.pressureBar      — давление, бар
CH.pressureCurrent  — ток датчика, мА
```

Авария датчика ставится, если:

```text
currentMa === null
или
currentMa < currentBreakMa
```

Текущее значение `currentBreakMa` — 3.6 мА.

При аварии датчика:

1. Ставится `sensorAlarm = true`.
2. Клапан закрывается.
3. Цикл подпитки останавливается.
4. Автоматическая подпитка не начинается.

## 3.3. Фильтрация давления

Давление фильтруется через `filterPressure(rawValue)`.

Принцип:

1. Новое значение добавляется в буфер.
2. Буфер ограничивается `CFG.filterSize`.
3. Считается среднее.
4. Результат ограничивается физическим диапазоном 0…6 бар.

Фильтр нужен, чтобы не запускать подпитку от одиночных скачков давления.

## 3.4. Старт автоматического цикла

Цикл начинается только если одновременно:

```text
enabled = true
auto_mode = true
sensor_alarm = false
pressureBar < pressureMinBar
STATE.active = false
```

При старте:

```text
STATE.active = true
STATE.pulseCount = 0
last_event = "Старт цикла подпитки"
```

## 3.5. Импульс подпитки

Импульс выполняет `openPulse(settings, manualMode)`.

Последовательность:

1. Проверить, что нет аварии датчика.
2. Проверить, что клапан не открыт.
3. Проверить, что не идёт пауза.
4. Установить `STATE.active = true`.
5. Установить `STATE.valveOpening = true`.
6. Увеличить `pulseCount`.
7. Открыть клапан: `dev[CH.valveMakeup] = true`.
8. Запустить `valveCloseTimer`.
9. Запустить `watchdogTimer`.

## 3.6. Закрытие после импульса

После `pulseOpenS`:

1. Клапан закрывается.
2. Сбрасывается `valveOpening`.
3. Ставится `waitingPause = true`.
4. Запускается `pauseTimer`.

## 3.7. Пауза после импульса

Во время паузы новые импульсы не запускаются.

После `pulsePauseS`:

1. Сбрасывается `waitingPause`.
2. Вызывается `evaluate()`.
3. Скрипт проверяет давление.

## 3.8. Продолжение цикла

Если цикл уже активен:

```text
STATE.active = true
pressureBar < pressureTargetBar
pulseCount < maxPulses
```

скрипт запускает следующий импульс.

Важно: цикл продолжается до `pressureTargetBar`, а не только до `pressureMinBar`.

## 3.9. Успешное завершение

Если:

```text
pressureBar >= pressureTargetBar
```

то:

1. Клапан закрывается.
2. Таймеры сбрасываются.
3. `STATE.active = false`.
4. `STATE.pulseCount = 0`.
5. `makeupFailedAlarm = false`.
6. Пишется событие успешного завершения.

## 3.10. Неуспешное завершение

Если:

```text
pulseCount >= maxPulses
pressureBar < pressureTargetBar
```

то:

1. Ставится `makeupFailedAlarm = true`.
2. Клапан закрывается.
3. Цикл останавливается.
4. Отправляется аварийное SMS, если разрешено `smsAllowed`.

# 4. Защиты

## 4.1. Закрытие на старте

Скрипт закрывает клапан при запуске.

Это защита от ситуации:

```text
клапан был открыт
wb-rules перезапустился
таймер закрытия потерялся
клапан остался открыт
```

## 4.2. Раздельные таймеры

В скрипте должны быть отдельные таймеры:

| Таймер | Назначение |
|---|---|
| `valveCloseTimer` | Закрытие клапана после импульса |
| `pauseTimer` | Пауза после закрытия клапана |
| `watchdogTimer` | Аварийное закрытие клапана |

Один общий таймер использовать нельзя. Это было причиной риска в старой версии.

## 4.3. Watchdog клапана

Если клапан открыт дольше `watchdogS`, watchdog:

1. Ставит `watchdogAlarm = true`.
2. Закрывает клапан.
3. Останавливает цикл.
4. Пишет событие.
5. Отправляет аварийное уведомление, если SMS разрешены.

## 4.4. Запрет повторного импульса

Новый импульс не запускается, если:

```text
sensorAlarm = true
valveOpening = true
waitingPause = true
A04/K1 уже открыт
```

## 4.5. Ограничение maxPulses

Скрипт не должен подпитывать бесконечно.

Если заданное количество импульсов не помогло, это уже авария:

```text
makeupFailedAlarm = true
```

## 4.6. Блокировка вторичных тревог

Канал:

```text
power_monitor/secondary_alerts_blocked
```

используется для SMS.

Если блокировка активна, функция `sendAlertIfAllowed(...)` не отправляет SMS.

Логика управления клапаном от этого не зависит: блокировка вторичных тревог отключает уведомления, но не должна сама по себе открывать или закрывать клапан.

# 5. Что файл не делает

Файл не выполняет следующие функции:

1. Не управляет насосами.
2. Не управляет котлом.
3. Не управляет OpenTherm.
4. Не контролирует физическое положение клапана, если нет обратной связи.
5. Не измеряет расход воды.
6. Не отличает утечку от заполнения системы воздухом.
7. Не ведёт долговременную историю.
8. Не отправляет e-mail.
9. Не принимает решения по режиму «приехали / уехали».

# 6. Предложения по улучшению

## 6.1. Дебаунс аварии датчика

Сейчас отсутствие тока датчика сразу считается аварией.

На некоторых объектах после рестарта `wb-rules` значение тока может появиться не мгновенно. Возможное улучшение:

```text
если currentMa === null или currentMa < 3.6 мА
→ подождать 10–30 секунд
→ проверить повторно
→ только потом ставить sensor_alarm
```

Это снизит ложные аварии после перезапуска.

## 6.2. Авария канала давления

Сейчас авария датчика завязана на ток. Если ток нормальный, но канал давления `pressureBar` отсутствует, подпитка не начнётся, но явной аварии может не быть.

Предложение:

```text
если currentMa нормальный, но pressureBar === null
→ поставить отдельный флаг ошибки пересчёта давления
```

## 6.3. Контроль роста давления

Сейчас скрипт смотрит только итоговое давление после импульсов.

Дополнительная диагностика:

```text
если за несколько импульсов давление почти не растёт
→ авария "подпитка не влияет на давление"
```

Это поможет отличать:

- закрытый шаровой кран перед клапаном;
- отсутствие воды;
- неисправный клапан;
- сильную утечку;
- воздух в системе.

## 6.4. Обратная связь клапана

Если на будущем объекте есть вход контроля клапана или давления ХВС, добавить проверку:

```text
команда клапану дана
а фактического признака открытия нет
```

Сейчас такой обратной связи нет.

## 6.5. Ограничение общего времени цикла

Кроме `maxPulses`, можно добавить общий лимит времени цикла:

```text
cycleMaxTimeS
```

Это дополнительная защита от нештатных состояний.

## 6.6. Отдельное уведомление по низкому давлению

Сейчас `low_pressure_alarm` отображается в виртуальном устройстве.

Предложение:

```text
если pressureBar < pressureAlarmBar
→ отправить отдельное аварийное SMS
```

Но лучше включать это после проверки, чтобы не получить спам при ПНР.

## 6.7. Подтверждение восстановления

Можно отправлять восстановление:

```text
ВОССТАНОВЛЕНО: давление отопления восстановлено
```

когда аварийное состояние ушло.

Сейчас основная задача — безопасная подпитка, поэтому восстановительные SMS можно добавлять позже.

## 6.8. Логирование в отдельный файл

Сейчас используется `safeLog()`, то есть сообщения уходят в журнал `wb-rules`.

В будущем можно сделать отдельный механизм отчёта:

```text
события подпитки
количество импульсов
давление до / после
аварии
```

Это полезно для еженедельного отчёта на e-mail.

6. Проверить, что `manual_close` закрывает клапан во время импульса.

# 8. Порядок первого теста на объекте

## 8.1. Перед тестом

```bash
grep -R "A04/K1" /etc/wb-rules /etc/wb-rules-modules
```

Должен быть только один писатель в клапан подпитки.

Проверить текущее состояние клапана:

```bash
mosquitto_sub -v -t '/devices/A04/controls/K1' -C 1 -W 3
```

Ожидаемо:

```text
/devices/A04/controls/K1 0
```

## 8.2. После установки файла

```bash
systemctl restart wb-rules
journalctl -u wb-rules -f | grep -E "507_Pressure|pressure_makeup|rule error|TypeError|ReferenceError|SyntaxError"
```

Проверить, что при старте есть событие:

```text
Принудительное закрытие клапана на старте
```

## 8.3. Тест ручного импульса

1. `enabled = false`.
2. `auto_mode = false`.
3. Нажать `manual_pulse`.
4. Проверить, что клапан открылся.
5. Проверить, что клапан закрылся через `pulse_open_s`.
6. Проверить, что watchdog не сработал.

## 8.4. Тест ручного закрытия

1. Нажать `manual_pulse`.
2. Пока клапан открыт, нажать `manual_close`.
3. Проверить, что `A04/K1 = 0`.
4. Проверить `last_event`.

## 8.5. Тест аварии датчика

Вариант безопасной имитации зависит от объекта.

Проверить:

```text
если ток датчика пропал или стал ниже 3.6 мА
→ клапан закрыт
→ sensor_alarm = true
→ новый импульс не запускается
```

## 8.6. Тест автоматического режима

1. Установить безопасные уставки.
2. Включить `enabled`.
3. Включить `auto_mode`.
4. Спровоцировать давление ниже `pressureMinBar`.
5. Наблюдать весь цикл.
6. Не уходить от объекта до завершения теста.

# 9. Признаки нормальной работы

Нормальный цикл выглядит так:

```text
Старт цикла подпитки
Открытие клапана
Закрытие клапана
Начало паузы
Окончание паузы
...
Успешное завершение: достигнуто целевое давление
```

Виртуальные поля:

| Поле | Нормальное состояние после завершения |
|---|---|
| `valve_open` | `false` |
| `active` | `false` |
| `valve_opening` | `false` |
| `waiting_pause` | `false` |
| `sensor_alarm` | `false` |
| `makeup_failed_alarm` | `false` |
| `watchdog_alarm` | `false` |
| `pressure_bar` | >= `pressureTargetBar` |

# 10. Признаки проблем

| Признак | Возможная причина |
|---|---|
| `valve_open = true` дольше `pulseOpenS` | Не сработал таймер, зависло реле, ошибка скрипта |
| Сработал `watchdog_alarm` | Клапан не закрылся вовремя |
| `sensor_alarm = true` | Нет тока датчика или ток ниже 3.6 мА |
| Давление не растёт после импульсов | Нет воды, закрыт кран, неисправен клапан, утечка, воздух |
| `makeup_failed_alarm = true` | Достигнут `maxPulses`, давление не поднялось до цели |
| Подпитка не стартует | `enabled=false`, `auto_mode=false`, авария датчика, давление не ниже минимума |
| Цикл активен, но клапан не открывается | Идёт пауза, достигнут maxPulses, ошибка состояния |
| SMS не пришло | Заблокированы вторичные тревоги, проблема модема, длинное сообщение |

# 11. Рекомендации для другой ветки ИИ

При доработке файла не менять архитектуру без причины.

Обязательные правила:

1. Не использовать `sleep()`.
2. Не использовать `let`, `const`, стрелочные функции.
3. Не писать в `CH.valveMakeup` из другого файла.
4. Не объединять таймеры в один общий `STATE.timer`.
5. Не запускать новый импульс во время `valveOpening`.
6. Не запускать новый импульс во время `waitingPause`.
7. Не считать цикл успешным по `pressureMinBar`; успешное завершение только по `pressureTargetBar`.
8. Не продолжать подпитку после `maxPulses`.
9. Не открывать клапан при аварии датчика.
10. При рестарте скрипта клапан должен закрываться.
11. Все изменения писать полным файлом.
12. Комментарии внутри файла — короткие, на русском.

Если есть сомнение по каналу или физике объекта, не придумывать. Писать:

```text
НЕ ПРОВЕРЕНО: требуется уточнить ...
```

# 12. Краткое описание для GitHub

`507_Pressure_makeup.js` — скрипт автоматической импульсной подпитки давления отопления.

Функции:

- контроль давления отопления;
- контроль тока датчика 4–20 мА;
- импульсное открытие клапана подпитки;
- пауза после каждого импульса;
- продолжение цикла до целевого давления;
- ограничение по количеству импульсов;
- watchdog аварийного закрытия клапана;
- принудительное закрытие клапана на старте;
- ручной импульс;
- ручное закрытие;
- сброс аварий;
- диагностическое виртуальное устройство `pressure_makeup`.

Критичные защиты:

- клапан закрывается при старте скрипта;
- клапан закрывается при ошибке датчика;
- клапан закрывается при ручном отключении;
- новый импульс не запускается во время открытого клапана или паузы;
- после `maxPulses` подпитка останавливается;
- watchdog закрывает клапан, если он открыт дольше допустимого времени.

Файл требует обязательной проверки каналов перед переносом на другой объект.
