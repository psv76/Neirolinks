## 1. Цель архитектуры
Архитектура должна обеспечивать:

- разделение общесистемных функций и логики конкретных подсистем
- повторное использование модулей на других объектах
- минимизацию дублирования кода
- удобную диагностику
- безопасное развитие проекта без превращения одного файла в “монолит”

Для этого вся логика делится на два уровня:

Общие модули
Каталог:

/etc/wb-rules-modules/
Сценарии объекта
Каталог:

/etc/wb-rules/
## 2. Принцип разделения
### 2.1. Что должно лежать в wb-rules-modules
Только то, что одинаково используется несколькими сценариями:

- общие настройки объекта

- Telegram

- форматирование сообщений

- общие функции таймеров и диагностики

- общие функции чтения и нормализации значений

- общие функции журналирования

### 2.2. Что должно лежать в wb-rules
Только логика конкретных инженерных подсистем объекта:

- мониторинг электроснабжения

- мониторинг основного питания через UPS

- мониторинг температуры

- логика отопления

- логика освещения

- логика сервоприводов

- графические и сервисные виртуальные устройства

## 4. Назначение каждого файла
### 4.1. system.js
Главный общий модуль объекта.

Содержит:

- имя объекта

- имя строения

- общие флаги

- общие параметры Telegram

- общие функции сборки сообщений

- экспорт конфигурации для остальных сценариев

В этом файле должны лежать только глобальные параметры объекта, например:

- objectName

- buildingName

- Telegram token

- Telegram chatId

- признак резервирования роутера

пути до общих статусов объекта

Не должны лежать:

- автоматы конкретных линий

- контакторы конкретных нагревателей

- каналы конкретной подсистемы

Это важно:
system.js — это инфраструктурный модуль, а не конфиг всей автоматики.

### 4.2. telegram.js
Сервис уведомлений.

Назначение:

- отправка сообщений в Telegram

- единое место обработки ошибок отправки

- единый формат текста

в будущем — маршрутизация по разным чатам или уровням тревог

Функции:

- sendMessage(text)

- sendAlert(text, recommendation)

- sendService(text)

Если оставить отправку Telegram только в system.js, модуль будет перегружен.
Поэтому в конечной архитектуре правильнее отделить:

system.js — конфигурация

telegram.js — транспорт сообщений

### 4.3. utils.js
Набор общих утилит.

Содержит:

- преобразование значений в bool

- безопасное чтение dev[...]

- проверку списка каналов на наличие активного сигнала

- функции нормализации значений

- возможно — форматирование инженерных строк

Примеры задач:

readBool(path)

readNumber(path, fallback)

anyOn(list)

safeString(value)

### 4.4. timers.js
Общий модуль работы с таймерами и таймерными состояниями.

Нужен потому, что в объекте уже есть и дальше будут:

- блокировка уведомлений

- задержка после отключения питания

- задержка после восстановления питания

- задержки для проверки нагревателей

- задержки по температурным авариям

Лучше не разносить эту механику по каждому сценарию отдельно.

Содержит:

- запуск таймера

- отмену таймера

- хранение таймеров по ключу

- функции проверки активного окна

### 4.5. diagnostics.js
Единый модуль диагностического логирования.

Назначение:

- единообразные сообщения в лог

- включение/выключение подробного лога

- упрощение диагностики на пусконаладке

Функции:

diagInfo(text)

diagWarn(text)

diagError(text)

## 5. Скрипты подсистем объекта
### 5.1. 00_power_monitor.js
Главный сценарий мониторинга электроснабжения объекта.

Это приоритетный сценарий, потому что от него зависит корректность всех остальных уведомлений.

Функции:

- контроль основного питания щита через WB-UPS

- подавление частных тревог при потере ввода щита

- контроль освещения

- контроль сервоприводов

- контроль обогревателей

- контроль дифавтоматов

- контроль контакторов

- уведомления о пропадании питания

- уведомления о восстановлении

- блокировка повторных уведомлений

- повторная проверка после окончания блокировки

- виртуальное устройство Мониторинг электроснабжения v2.0

Это верхнеуровневый сценарий, который влияет на поведение остальных ветвей.

### 5.2. 01_temperature_monitor.js
Сценарий мониторинга температуры.

Назначение:

- контроль температуры воздуха

- контроль температуры грядок

- контроль температуры дорожки

- контроль уставок

- задержки перед аварийными уведомлениями

- сообщения о выходе за пределы

- сообщения о возврате в норму

Он должен быть отдельным, а не смешанным с электроснабжением.

### 5.3. 02_heating_control.js
Сценарий управления отоплением.

Назначение:

- логика включения котла

- логика открытия контуров

- логика работы ИК-обогревателей

- выбор режима отопления

- связь с уставками температуры

- реакция на погодные условия

- подготовка команд на реле и сервоприводы

Важно:
это управляющая логика, а не мониторинг.

### 5.4. 03_lighting_control.js
Сценарий управления освещением.

Назначение:

- расписание фитосвета

- расписание основного света

- ручной режим

- сервисный режим

- возможные сезонные настройки

Мониторинг питания света остаётся в power_monitor.js,
а логика включения света — здесь.

### 5.5. 04_servo_control.js
Сценарий управления сервоприводами.

Назначение:

- управление сервоприводами контуров

- режимы открытия/закрытия

- безопасные состояния

- логика при отказах и ручном режиме

### 5.6. 90_service_ui.js
Сервисный интерфейс и вспомогательные виртуальные устройства.

Назначение:

- служебные кнопки

- тестовые сценарии

- переключатели сервисных режимов

- диагностические индикаторы

- интерфейсные группировки

Если не отделить сервисный UI, интерфейсные элементы начнут расползаться по функциональным сценариям.

## 7. Главные инженерные правила для этого объекта
### 7.1. Мониторинг и управление не смешивать
Нельзя помещать в один сценарий:

управление освещением по расписанию

и контроль его электроснабжения

Это две разные ответственности.

Правильно:

lighting_control.js — включает свет

power_monitor.js — проверяет, появился ли на линии реальный ток/питание

### 7.2. Корневая причина должна иметь приоритет
Если пропадает основное питание щита, это событие должно быть выше частных событий.

Поэтому 00_power_monitor.js должен:

сначала определить mains_fail

затем подавить частные тревоги

после восстановления снова разрешить частную диагностику

Это главный принцип v2.0.

### 7.3. Общий модуль не должен превращаться в “свалку конфигов”
В system.js не должны попадать:

автоматы освещения

контакторы обогревателей

каналы отдельных подсистем

Это принадлежит соответствующим сценариям.

### 7.4. Все виртуальные устройства должны проектироваться отдельно от бизнес-логики
Интерфейс — это не логика.
Логика должна жить в функциях и сценариях, а UI только отображать и управлять параметрами.

## 8. Архитектура уведомлений для этого объекта
### 8.1. Типы сообщений
На объекте должны быть минимум четыре класса сообщений:

Критические

пропало основное электроснабжение щита

Частные аварии

освещение

сервоприводы

обогреватели

Восстановление

восстановлено основное питание

восстановлено питание частной линии

Сервисные

тест

информация для пусконаладки

### 8.2. Кто формирует сообщения
telegram.js — отправляет

system.js — даёт объект и строение

конкретный сценарий — формирует смысл сообщения

То есть текст частной аварии по обогревателю должен оставаться в power_monitor.js, а не в общем модуле.

## 9. Архитектура состояния объекта
Для этого объекта нужно хранить три класса состояния.

### 9.1. Постоянная конфигурация
Хранится в коде:

каналы

имена аппаратов

имена зон

идентификаторы устройств

### 9.2. Оперативное состояние
Хранится в переменных сценариев:

активная авария

pending-состояние

флаги блокировки

таймеры

mainsFailActive

### 9.3. Пользовательские настройки
Хранятся через виртуальные устройства:

переключатели мониторинга

задержки

таймеры

тестовые функции

## 13. Итог
Для объекта “Теплица” правильная архитектура скриптов должна быть следующей:

общие модули вынесены в /etc/wb-rules-modules

логика подсистем разнесена по отдельным сценариям в /etc/wb-rules

power_monitor является верхнеуровневым сценарным слоем

основное питание щита имеет приоритет над частными тревогами

температура, отопление, свет, сервоприводы должны быть отдельными сценариями

виртуальные устройства и сервисный UI должны быть вынесены в отдельный сервисный слой

Это и есть оптимальная инженерная архитектура скриптов для данного объекта.
