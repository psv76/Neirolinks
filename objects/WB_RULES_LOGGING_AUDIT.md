## 3. Детальный разбор по объектам

## Объект 05_31_Ivolga_13

### `objects/05_31_Ivolga_13/etc/wb-rules-modules/system.js`

- Назначение: общий модуль `system` для SMS и формирования сообщений; экспортирует `safeLog`, `sendAlert`, `sendRestore`, `sendInfo`.
- Критичность: A, потому что используется аварийными уведомлениями.
- Логирование: есть. `safeLog(message)` вызывает `log(String(message))` в `try` (`system.js:10-13`). SMS-ветки пишут `[system] SMS отключены`, `[system] Попытка отправки SMS...`, `[system] Ошибка Notify.sendSMS...` (`system.js:63-89`).
- Префикс: `[system]`; для фильтрации модуля достаточен, но не содержит объект и вызывающий скрипт.
- Виртуальное устройство: нет.
- Реальные выходы: нет.
- Диагностические каналы: нет.
- События, которые уже логируются: состояние отправки SMS, пустые номер/сообщение, попытка и ошибка `Notify.sendSMS`.
- События, которых не хватает: единое journal-событие при `sendAlert/sendRestore/sendInfo` до/после отправки, код события, источник вызывающего скрипта.
- Риски: в журнале виден `[system]`, но без контекста правила, инициировавшего аварию.
- Комментарий для будущего стандарта: модуль подходит как точка для общей функции `logEvent(object, script, code, severity, text, details)`.

## 4. Повторяющиеся проблемы

- Нет единого префикса. Используются `[system]`, `[100_power_monitor]`, `[507_Pressure_makeup]`, `[Kitchen_Hood]`, `[550_Pressure_makeup]`, но не формат `[объект][скрипт]`.
- События не имеют кодов: нет `START`, `CMD`, `STATE`, `ALARM`, `RECOVERY`, `WATCHDOG`, `OUTPUT_ERROR` как машинно-фильтруемых кодов.
- `last_event` перезаписывается и не дает истории; в стандартных WB-rules нет `event_counter/event_code/event_time`.
- Реальные выходы часто пишутся напрямую через `dev[...] = ...` без проверки существования канала и без повторного чтения.
- Стартовые `evaluate()`/apply выполняются через 1-3 секунды после загрузки: `250_Gate.js` читает ворота через 1 с, `260_Kitchen_Vent.js` может писать выходы через 1 с, `100_power_monitor.js` и `600_Heat_diagnostics.js` пишут VDev через 3 с, `507/550_Pressure_makeup.js` закрывают клапан через 3 с.
- Ошибки `failed to SetValue for unexisting control` могут потеряться в шуме системного журнала, потому что сами скрипты не пишут `OUTPUT_ERROR`.
- Нет единообразного разделения `manual`/`auto`/`startup`. В подпитке ручной импульс текстом отличается, но машинного поля нет.
- Нет снапшота параметров в момент события: давление, ток, входы, целевой уровень, активные датчики остаются только в текущих VDev-полях.
- Нет явной настройки хранения `event_*` каналов в истории; в самих скриптах `event_*` почти отсутствуют.
- Нет единого словаря кодов событий и уровней важности.
- Для `System.sendAlert/sendRestore/sendInfo` нет обязательной записи в journal с источником вызывающего скрипта.

## 5. Что нужно для стандарта логирования

По фактам найденных скриптов нужен стандарт, покрывающий такие типы событий:

- `START`: запуск скрипта; найдено в `100_power_monitor`, `507_Pressure_makeup`, `550_Pressure_makeup`, но не единообразно.
- `READY`: скрипт готов после стартовой задержки/evaluate; сейчас явно не найдено.
- `BLOCK`: блокировка вторичных тревог в `100_power_monitor` и блокировки подпитки.
- `CMD`: команда на исполнительный канал: клапан, насос, реле света, ворота, диммер, 0-10 В вентиляции.
- `STATE`: изменение режима: auto/manual, цикл активен, пауза, фон/скорость вентиляции, приоритет ГВС.
- `ALARM`: авария питания, UPS, датчика давления, низкого давления, watchdog клапана, перегрева/котла/циркуляции, вентиляции.
- `RECOVERY`: восстановление питания, UPS, датчиков, давления, снятие аварии.
- `MANUAL`: ручной импульс подпитки, ручное закрытие, нажатия кнопок/мастер-сцен.
- `AUTO`: автоматический старт подпитки, автоматическая установка вентиляции, отопительные насосы.
- `WATCHDOG`: зависание клапана подпитки; потенциально таймеры вентиляции/ворот.
- `LIMIT`: максимум импульсов, аварийный минимум давления, пределы яркости, лимиты температур/ΔT.
- `SENSOR`: обрыв/невалидные датчики давления, температуры, токов, OpenTherm/UPS.
- `OUTPUT_ERROR`: недоступен исполнительный канал или запись не подтверждена.

Минимальный набор MQTT-полей для критичных скриптов:

- `event_time` — unix timestamp или ISO-строка;
- `event_counter` — монотонный счетчик событий;
- `event_code` — машинный код;
- `event_severity` — info/warning/error/alarm;
- `event_text` — человекочитаемый текст;
- `event_source` — имя скрипта;
- `event_object` — объект;
- профильные поля: `event_pressure_bar`, `event_current_ma`, `event_output`, `event_target_state`, `event_reason`, `event_mode`.

Формат journal-сообщения желательно нормализовать:

```text
[05_38_Plod_47][550_Pressure_makeup][ALARM] Низкое давление отопления; heat=0.92 bar; cold=3.1 bar; auto=true
```
