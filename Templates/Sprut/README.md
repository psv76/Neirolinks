# NEIROLINKS Sprut.hub MQTT Templates

Папка содержит шаблоны Sprut.hub для виртуальных термостатов, которые создаются в Wiren Board через `wb-rules` и передаются в Sprut.hub по MQTT.

Назначение шаблонов — чтобы Sprut.hub автоматически распознавал термостаты NEIROLINKS по имени MQTT-устройства, создавал правильный аксессуар и передавал в WB уставки, режимы и настройки.

## Общий принцип

Логика отопления и климата работает в Wiren Board.

Sprut.hub используется как интерфейс:

- отображение температуры;
- изменение уставки;
- включение / выключение термостата;
- выбор режима;
- передача дополнительных настроек в WB.

Критичная логика не должна переноситься в сценарии Sprut.hub.

## Как Sprut.hub распознаёт устройство

Шаблон ищет MQTT-устройство по имени через `modelIds`.

Пример:

```json
"modelIds": [
  "/devices/(NL_combo_thermostat_[0-9]{3})/controls/air_temperature/meta/type"
]
```

Это значит:

- устройство в WB должно называться по маске `NL_combo_thermostat_607`;
- номер в конце должен состоять из трёх цифр;
- у устройства должен быть канал `air_temperature`.

`[0-9]{3}` — это регулярное выражение: любые три цифры.

Подойдут:

```text
NL_combo_thermostat_607
NL_combo_thermostat_001
NL_combo_thermostat_999
```

Не подойдут:

```text
NL_combo_thermostat_7
NL_combo_thermostat_607_608
heating_child_right_thermostat
```

## Принятая схема имён

Виртуальные устройства WB должны называться так:

```text
NL_simple_thermostat_607
NL_combo_thermostat_607
NL_climate_thermostat_607
```

Где `607` — проектный номер, выбранный для этой зоны.

Если термостат управляет несколькими физическими каналами, в имени всё равно используется один выбранный номер. Полный список физических каналов указывается в WB-скрипте и в системном журнале.

Пример:

```text
Имя виртуального устройства:
NL_combo_thermostat_607

Физические каналы в WB-скрипте:
wb-mr6cu_224/K1
wb-mr6cu_224/K2
```

## Шаблоны

### `NL_simple_thermostat.json`

Простой термостат.

Используется, когда у зоны один датчик температуры и одна команда нагрева.

Примеры применения:

- радиаторная зона с датчиком воздуха и сервоприводом радиатора;
- простой контур тёплого пола с датчиком пола;
- любая простая зона, где не нужна отдельная логика `воздух + пол`.

Имя виртуального устройства WB:

```text
NL_simple_thermostat_607
```

Маска распознавания:

```text
NL_simple_thermostat_[0-9]{3}
```

Обязательные каналы виртуального устройства:

```text
temperature
target_temperature
target_state
current_state
```

Режимы:

```text
OFF
HEAT
```

Смысл каналов:

| Канал | Назначение |
|---|---|
| `temperature` | текущая температура |
| `target_temperature` | целевая температура |
| `target_state` | включение / выключение термостата |
| `current_state` | фактический текущий режим нагрева |

---

### `NL_combo_thermostat.json`

Комбинированный термостат для водяного тёплого пола.

Используется, когда у зоны есть два датчика:

- температура воздуха;
- температура пола.

И одна итоговая команда нагрева зоны:

- один сервопривод;
- или несколько сервоприводов, работающих как одна зона.

Основной сценарий — дом без радиаторов, где воздух нагревается тёплым полом.

Имя виртуального устройства WB:

```text
NL_combo_thermostat_607
```

Маска распознавания:

```text
NL_combo_thermostat_[0-9]{3}
```

Обязательные каналы виртуального устройства:

```text
air_temperature
floor_temperature
target_temperature
target_state
current_state
floor_min_temperature
floor_max_temperature
```

Режимы:

```text
OFF
HEAT
```

Смысл каналов:

| Канал | Назначение |
|---|---|
| `air_temperature` | текущая температура воздуха |
| `floor_temperature` | текущая температура пола |
| `target_temperature` | уставка воздуха |
| `target_state` | включение / выключение термостата |
| `current_state` | фактическая команда нагрева зоны |
| `floor_min_temperature` | цель пола при поддержании |
| `floor_max_temperature` | цель пола при нагреве комнаты |

Логика WB для такого термостата:

```text
если воздух ниже уставки:
    цель пола = Пол при нагреве

если воздух в норме:
    цель пола = Пол при поддержании

сервоприводы открываются/закрываются по температуре пола относительно выбранной цели
```

---

### `NL_climate_thermostat.json`

Климатический термостат.

Используется, когда зона управляет не только отоплением, но и охлаждением.

Радиатор не обязателен.

Минимальный вариант:

```text
воздух + пол
нагрев через тёплый пол
охлаждение через кондиционер
```

Расширенный вариант:

```text
воздух + пол
нагрев через тёплый пол
нагрев через радиатор
охлаждение через кондиционер
```

Имя виртуального устройства WB:

```text
NL_climate_thermostat_607
```

Маска распознавания:

```text
NL_climate_thermostat_[0-9]{3}
```

Обязательные каналы виртуального устройства:

```text
air_temperature
floor_temperature
target_temperature
target_state
current_state
floor_min_temperature
floor_max_temperature
cooling_temperature
```

Режимы:

```text
OFF
HEAT
COOL
AUTO
```

Смысл каналов:

| Канал | Назначение |
|---|---|
| `air_temperature` | текущая температура воздуха |
| `floor_temperature` | текущая температура пола |
| `target_temperature` | основная уставка воздуха |
| `target_state` | целевой режим термостата |
| `current_state` | фактический текущий режим |
| `floor_min_temperature` | цель пола при поддержании |
| `floor_max_temperature` | цель пола при нагреве комнаты |
| `cooling_temperature` | порог / уставка охлаждения |

Ожидаемые значения `target_state`:

| Значение | Режим |
|---|---|
| `0` | OFF |
| `1` | HEAT |
| `2` | COOL |
| `3` | AUTO |

Ожидаемые значения `current_state`:

| Значение | Режим |
|---|---|
| `0` | OFF |
| `1` | HEAT |
| `2` | COOL |

## Как выбирать шаблон

| Ситуация | Шаблон |
|---|---|
| Один датчик температуры и одна команда нагрева | `NL_simple_thermostat` |
| Воздух + пол, только отопление через пол | `NL_combo_thermostat` |
| Воздух + пол + кондиционер | `NL_climate_thermostat` |
| Воздух + пол + радиатор + кондиционер | `NL_climate_thermostat` |

## Что должно создать WB-правило

WB-скрипт должен создать виртуальное устройство через `defineVirtualDevice(...)`.

Пример имени:

```javascript
var VDEVICE = "NL_combo_thermostat_607";
```

Пример заголовка:

```javascript
defineVirtualDevice(VDEVICE, {
    title: "Термостат детская справа",
    cells: {
        ...
    }
});
```

Важно различать:

| Поле | Смысл |
|---|---|
| `VDEVICE` | техническое MQTT-имя устройства |
| `title` | человекочитаемый заголовок в WebUI |
| `name` в шаблоне Sprut.hub | базовое имя аксессуара в Sprut.hub |

## Правило по номерам

Номер в имени выбирается по проекту.

Примеры:

```text
NL_combo_thermostat_607
NL_combo_thermostat_608
NL_climate_thermostat_615
```

Если зона управляет несколькими сервоприводами, номер в имени не обязан перечислять все каналы.

Правильно:

```text
NL_combo_thermostat_607
```

Неправильно:

```text
NL_combo_thermostat_607_608
```

Список фактических каналов должен быть внутри WB-скрипта:

```javascript
valveTopics: [
    "wb-mr6cu_224/K1",
    "wb-mr6cu_224/K2"
]
```

И в системном журнале:

```text
[отопление][610_thermostats][детская справа]; КОМАНДА=Открыть сервоприводы зоны; канал=wb-mr6cu_224/K1, wb-mr6cu_224/K2; значение в канал=ON
```

## Sprut.hub и Алиса

Шаблоны должны передавать в Sprut.hub нормальные типы устройств:

- `Thermostat` — основной термостат;
- `TemperatureSensor` — дополнительная температура пола.

Для Алисы важно, чтобы термостат выглядел как обычное климатическое устройство:

```text
текущая температура
целевая температура
режим нагрева / охлаждения
```

Для `NL_combo_thermostat` основной режим:

```text
OFF / HEAT
```

Для `NL_climate_thermostat` основной режим:

```text
OFF / HEAT / COOL / AUTO
```

## Важные правила для ИИ-ассистента

Если ИИ-ассистент создаёт новый WB-скрипт под эти шаблоны, он должен:

1. Выбрать тип термостата по задаче:
   - `simple`;
   - `combo`;
   - `climate`.

2. Создать виртуальное устройство с именем по маске:

```text
NL_<type>_thermostat_607
```

3. Не использовать старые тестовые имена:

```text
heating_child_right_thermostat
```

4. Не создавать отдельный Sprut-шаблон под каждую комнату.

5. Не добавлять номер второго сервопривода в имя устройства.

6. Не переносить отопительную логику в Sprut.hub.

7. Проверить, что набор каналов виртуального устройства соответствует выбранному шаблону.

8. Если зона с кондиционером, использовать `NL_climate_thermostat`, даже если радиатора нет.
