# NEIROLINKS — система логирования WB-rules

Дата фиксации: 10.05.2026

## 1. Назначение

Логирование нужно для расследования аварий и проверки работы инженерных сценариев через стандартный системный журнал Wiren Board.

Журнал должен быстро отвечать на вопросы:

```text
какая система сработала;
какой скрипт принял решение;
к какой зоне / узлу относится событие;
что произошло;
почему это произошло;
какие значения были в момент события;
в какой физический канал была отправлена команда.
```

Основной механизм:

```text
WebUI → Системный журнал → сервис wb-rules.service
```

Дополнительно используются:

```text
История MQTT физических каналов
```

Например, факт открытия сервопривода проверяется по истории физического канала реле, а причина открытия — по системному журналу `wb-rules`.

`event_*` каналы не являются обязательной частью каждого скрипта. Их добавлять только там, где они реально помогают расследованию и не дублируют уже полезную историю физических каналов.

---

## 2. Формат строки в системном журнале

Базовый формат:

```text
[система][скрипт][контекст]; ТИП=событие; параметр=значение; параметр=значение
```

Пример для термостата:

```text
[отопление][607_thermostat][детская справа]; СОСТОЯНИЕ=Пол нагрет; причина=воздух ниже уставки, но пол уже нагрет; режим=нагрев воздуха; воздух=23.3; уставка=25; пол=28.1; цель_пола=28
```

Пример команды:

```text
[отопление][607_thermostat][детская справа]; КОМАНДА=Открыть сервоприводы зоны; причина=воздух ниже уставки, пол ниже цели нагрева; канал=wb-mr6cu_224/K1, wb-mr6cu_224/K2; значение в канал=ON
```

Пример ошибки:

```text
[отопление][607_thermostat][детская справа]; ОШИБКА=Нет датчика пола, работаю по воздуху; причина=нет корректного значения пола; воздух=23.3; уставка=25
```

Пример аварии:

```text
[отопление][607_thermostat][детская справа]; АВАРИЯ=Нет обоих датчиков, сервоприводы закрыты; причина=нет корректных значений воздуха и пола; канал=wb-mr6cu_224/K1, wb-mr6cu_224/K2; значение в канал=OFF
```

## 3. Почему заголовок в квадратных скобках

Квадратные скобки — это короткий заголовок события:

```text
[система][скрипт][контекст]
```

Он нужен для быстрого поиска в WebUI.

Писать строчными буквами:

```text
[отопление]
[питание]
[вентиляция]
[детская справа]
```

Причина: в системном журнале WebUI по умолчанию включён поиск с учётом регистра `Aa`. Строчные теги проще искать без дополнительного клика.

## 4. Система

Первый блок — инженерная система.

Примеры:

| Система | Где использовать |
|---|---|
| `отопление` | термостаты, коллекторы, насосы, подпитка, подмесы, котёл |
| `питание` | ввод, фазы, UPS, аварии электроснабжения |
| `вода` | давление, подкачка, краны, водоснабжение |
| `протечки` | датчики протечки, перекрытие воды |
| `вентиляция` | вытяжки, приточка, рекуператор, заслонки |
| `ворота` | ворота, калитка, концевики |
| `освещение` | групповые команды, наружный свет |
| `уведомления` | Telegram/SMS/служебные уведомления |

Второй блок — скрипт. Фактически это источник логики / узел управления.

Третий блок — контекст:

```text
комната
зона
контур
узел
щит
```

Для зональных термостатов контекст = помещение.

## 5. Тип события

Тип события пишется после заголовка как `ТИП=событие`.

Используемые типы:

| Тип | Уровень журнала | Когда использовать |
|---|---|---|
| `СКРИПТ` | `log.info()` | загрузка, стартовая инициализация |
| `СОСТОЯНИЕ` | `log.info()` | изменение состояния алгоритма |
| `НАСТРОЙКА` | `log.info()` | пользователь изменил важную настройку |
| `КОМАНДА` | `log.info()` | скрипт отправил команду в канал |
| `РЕЗУЛЬТАТ` | `log.info()` | технологический результат после команды |
| `ОШИБКА` | `log.warning()` | отказ, невалидный датчик, недоступный канал, но сценарий ещё безопасно обработан |
| `АВАРИЯ` | `log.error()` | состояние требует внимания или сценарий не может нормально работать |
| `WATCHDOG` | `log.error()` | аварийное принудительное действие таймера безопасности |

`ДАТЧИК` и `ОТКАЗ` как отдельные типы не использовать. Это частные случаи `ОШИБКА`.

## 6. Команда и результат

`КОМАНДА` означает попытку записи в исполнительный канал.

Не писать:

```text
клапан открыт
сервопривод открыт
насос включён
```

если нет обратной связи.

Писать:

```text
КОМАНДА=Открыть клапан
КОМАНДА=Открыть сервоприводы зоны
КОМАНДА=Включить насос
```

Формат параметров команды:

```text
канал=...; значение в канал=ON
```

Физический или технологический итог писать отдельным событием `РЕЗУЛЬТАТ` или `АВАРИЯ`, если есть чем его проверить.

Проверку исполнения делать только при наличии реальной обратной связи:

```text
концевик;
датчик тока;
контроль 230 В;
датчик положения;
давление;
температура;
расход;
напряжение.
```

## 7. Уровни WebUI

Стандартные уровни системного журнала использовать как бесплатную фильтрацию:

```text
info     — нормальная работа;
warning  — ошибки и отказы без немедленной опасности;
error    — аварии и watchdog.
```

Тогда в WebUI можно выбрать:

```text
Тип сообщений = error
```

и увидеть только аварии по всем скриптам `wb-rules`.

## 8. Параметры строки

Параметры писать после события через `;`.

Рекомендуемые имена параметров:

| Параметр | Смысл |
|---|---|
| `причина` | почему скрипт принял решение |
| `режим` | внутренний режим алгоритма |
| `воздух` | температура воздуха в момент события |
| `уставка` | целевая температура воздуха |
| `пол` | температура пола в момент события |
| `цель_пола` | выбранная цель пола |
| `пол_поддержание` | низкая цель пола |
| `пол_нагрев` | высокая цель пола |
| `канал` | физический канал или список каналов |
| `значение в канал` | значение, отправленное в канал |
| `настройка` | какая настройка изменена |

Не добавлять в каждую строку все возможные данные. Писать только то, что помогает расследовать конкретное событие.

## 9. История MQTT и дублирование

Фактическое переключение реального исполнительного устройства смотреть по истории физического канала:

```text
wb-mr6cu_224/K1
wb-mr6cu_224/K2
```

Если скрипт создаёт виртуальный канал с тем же смыслом, это может дать дубль истории:

```text
физический канал уже пишется;
виртуальный канал может начать писаться отдельно;
данные будут повторяться.
```

Поэтому виртуальные каналы добавлять только если они нужны для интерфейса, Sprut.hub или диагностики.

Для зонального термостата:

```text
температура воздуха / пола в виртуальном устройстве нужна для Sprut.hub;
историю сервоприводов смотреть по физическим каналам;
event_* каналы не добавлять без отдельной причины.
```

Для подпитки, котельной, питания и других критичных процессов `event_*` каналы допустимы, если они фиксируют снимок события, который нельзя удобно получить из физической истории.

---

## 10. Когда нужны event_* каналы

`event_*` каналы нужны не каждому скрипту.

Добавлять их для процессов, где важен снимок состояния в момент события:

```text
подпитка давления;
питание;
котёл;
насосы с технологическим результатом;
протечки;
клапаны с аварийной логикой.
```

Минимальный набор для критичного процесса:

```text
event_counter
event_time_s
event_type
event_text
```

Профильные поля добавлять только по смыслу:

```text
event_heat_pressure_bar
event_cold_pressure_bar
event_delta_bar
event_pulse_count
event_valve_command
```

`event_text` можно писать в историю, если объект это поддерживает и это реально удобно при расследовании.

## 11. Локальная функция логирования

Пример для одного скрипта:

```javascript
var SYSTEM_NAME = "отопление";
var SCRIPT_NAME = "607_thermostat";
var ZONE_NAME = "детская справа";

function formatParams(params) {
    var list = [];
    var k;

    if (!params) {
        return "";
    }

    for (k in params) {
        if (params.hasOwnProperty(k)) {
            if (params[k] !== undefined && params[k] !== null && params[k] !== "") {
                list.push(k + "=" + params[k]);
            }
        }
    }

    if (list.length === 0) {
        return "";
    }

    return "; " + list.join("; ");
}

function writeLog(eventName, eventText, params) {
    var message = "[" + SYSTEM_NAME + "][" + SCRIPT_NAME + "][" + ZONE_NAME + "]; " +
        eventName + "=" + eventText + formatParams(params);

    if (eventName === "АВАРИЯ" || eventName === "WATCHDOG") {
        log.error(message);
        return;
    }

    if (eventName === "ОШИБКА") {
        log.warning(message);
        return;
    }

    log.info(message);
}
```

Позже функцию можно вынести в общий модуль, если формат стабилизирован на нескольких объектах.

## 12. Правило для кодов событий

Числовые коды событий не являются обязательной частью текущего стандарта.

Причина: в WebUI расследование удобнее вести по:

```text
системе;
скрипту;
контексту;
типу события;
физическому каналу;
тексту причины.
```

Если конкретному критичному скрипту нужны числовые коды, таблицу кодов писать прямо в начале этого скрипта или в README объекта.

Не вводить глобальную таблицу кодов для всех скриптов без отдельного решения.

## 13. Зональные термостаты

Для зональных термостатов текущий формат:

```text
[отопление][607_thermostat][детская справа]; СОСТОЯНИЕ=...
```

Если зон несколько и логика одинаковая, использовать один скрипт с конфигурацией зон.

Для каждой зоны:

```text
отдельное виртуальное устройство;
свои датчики;
свои сервоприводы;
свой контекст в логе.
```

Физический запрос тепла зоны = открыт хотя бы один сервопривод этой зоны.

`current_state` для Sprut.hub должен отражать фактическую команду нагрева зоны, а не промежуточное желание воздуха нагреваться.
