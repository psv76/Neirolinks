# Wiren Board coding standard

Этот документ задаёт правила разработки для объектов на Wiren Board в этом проекте.

## 1. Общие принципы

- Код должен быть совместим с реальным `wb-rules`, а не с обычным JavaScript/Node.js.
- Всегда учитывать ограничения и особенности `wb-rules` до написания кода.
- При изменениях присылать **полный код файла целиком**, а не фрагменты и не патчи.
- Если есть сомнение, как именно работает механизм на WB, приоритет у практической совместимости с объектом.

## 2. Ограничения wb-rules

- Использовать стиль **ECMAScript 5**.
- Не использовать:
  - `let`
  - `const`, если есть риск несовместимости в конкретном окружении
  - стрелочные функции
  - современные конструкции JavaScript без проверки совместимости с `wb-rules`
- Предпочитать:
  - `var`
  - обычные `function (...) { ... }`
  - простые циклы `for`

## 3. Изоляция сценариев

- Файлы в `/etc/wb-rules` выполняются в отдельных контекстах.
- Нельзя рассчитывать, что переменная или функция из одного файла будет доступна в другом файле автоматически.
- Общие функции и библиотеки выносить в модули:
  - путь: `/etc/wb-rules-modules`
  - подключение: `require("ModuleName")`

## 4. Структура файлов

Предпочтительная структура для объектов:

- `/etc/wb-rules-modules/WBHelpers.js` — общие функции
- `/etc/wb-rules/10_DeltaTemp.js` — корректировки температур
- `/etc/wb-rules/20_AlarmRules.js` — уведомления и аварии

Если появляются новые типовые блоки логики, выносить их в отдельные понятные файлы, но без лишнего дробления.

## 5. Правила для общих модулей

- Повторяющийся код выносить в модуль.
- Модули должны быть пригодны для повторного использования на следующих объектах.
- Всё общее: форматирование текста, задержки, антидребезг, отправка уведомлений, общие проверки — выносить в helper-модули.
- В модулях экспортировать только реально нужные функции.

## 6. Правила для уведомлений

- Для Telegram использовать штатные возможности Wiren Board, если они доступны в системе.
- Предпочтительный способ отправки:
  - `Notify.sendTelegramMessage(...)`
- Логика уведомлений должна быть в `wb-rules`, если условия сложнее, чем простые пороги из стандартного модуля уведомлений.
- Сообщения должны быть эксплуатационными:
  - не просто “компрессор в аварии”
  - а **почему** он в аварии
- В тексте уведомлений указывать:
  - объект
  - зону / камеру / МО
  - узел
  - причину

## 7. Правила для аварий

- Если у узла несколько причин аварии, сообщения делать **по причине**, а не одной общей фразой.
- Восстановление отправлять только тогда, когда действительно сняты все причины данной аварии, если этого требует логика эксплуатации.
- Не допускать спама одинаковыми уведомлениями.
- Использовать задержки перед отправкой, если это соответствует реальной логике объекта.

## 8. Правила для температур

- Корректировки температуры делать единым шаблоном, а не набором однотипных файлов на каждую зону.
- Для новых зон желательно добавлять только строки конфигурации, а не переписывать логику.
- Температурные уведомления строить по скорректированной температуре, если это принятая логика объекта.

## 9. Работа с каналами

- Не придумывать каналы “по аналогии”, если можно опереться на реальные:
  - `wb-webui.conf`
  - исполнительную схему
  - фактическую карту сигналов
  - подтверждённые пользователем MQTT device id
- При конфликте между проектом и фактом приоритет у **фактического подключения на объекте**.
- Если карта каналов уже была установлена в чате, считать её рабочей и не возвращаться к старым предположениям.

## 10. Стиль подготовки решений

- Сначала добиваться совместимости с WB.
- Только потом думать о “красоте архитектуры”.
- Не делать абстрактные документы и описания ради описаний.
- Все документы и скрипты должны быть полезны для:
  - эксплуатации текущего объекта
  - повторного использования на следующем похожем объекте

## 11. Что считать хорошим результатом

Хорошее решение для этого проекта — это решение, которое:
- реально работает на `wb-rules`
- легко переносится на следующий объект
- использует повторно общие модули
- не требует переписывать десятки файлов при мелкой правке
- понятно по каналам, авариям и уведомлениям

## 12. Комментарии в шапке скрипта

В шапке писать только то, что поможет через год понять назначение файла.

Нужно:

```text
что делает скрипт;
какой объект / зона;
краткая логика работы;
какие реальные каналы пишет скрипт.
```

Не писать в шапке:

```text
историю обсуждений;
что было удалено;
какие варианты отвергнуты;
предупреждения очевидного характера;
длинные инструкции пользователю.
```

## 13. Логирование WB-rules

Для инженерных сценариев использовать системный журнал `wb-rules.service`.

Формат строки:

```text
[система][скрипт][контекст]; ТИП=событие; параметр=значение
```

Пример:

```text
[отопление][607_thermostat][детская справа]; КОМАНДА=Открыть сервоприводы зоны; причина=воздух ниже уставки, пол ниже цели нагрева; канал=wb-mr6cu_224/K1, wb-mr6cu_224/K2; значение в канал=ON
```

Правила уровней:

```text
СКРИПТ / СОСТОЯНИЕ / НАСТРОЙКА / КОМАНДА / РЕЗУЛЬТАТ → log.info()
ОШИБКА → log.warning()
АВАРИЯ / WATCHDOG → log.error()
```

Теги в квадратных скобках писать строчными буквами, чтобы поиск в WebUI работал без отключения `Aa`.

## 14. Команда, результат и обратная связь

`КОМАНДА` — это попытка записи в канал, а не доказательство физического выполнения.

Если нет обратной связи, писать:

```text
КОМАНДА=Открыть клапан
```

а не:

```text
Клапан открыт
```

Проверку выполнения добавлять только при наличии:

```text
концевика;
датчика тока;
контроля 230 В;
датчика положения;
технологического параметра.
```

Не усложнять каждый сценарий проверками выполнения без реальной обратной связи.

## 15. История MQTT и виртуальные каналы

Не создавать виртуальные дубли физических каналов только ради удобства.

Если физический канал уже пишется в историю, дублирующий виртуальный канал будет создавать вторую историю с тем же смыслом.

Виртуальные каналы создавать, когда они нужны для:

```text
Sprut.hub;
понятного интерфейса WB;
сводного состояния;
настроек;
диагностики, которую нельзя удобно получить из физических каналов.
```

`event_*` каналы добавлять только для критичных процессов, где нужен снимок состояния в момент события.

## 16. Зональные термостаты

Если зон много и логика одинаковая, делать один скрипт с конфигурацией зон, а не набор однотипных файлов.

Для новых зон желательно добавлять только строку конфигурации.

Для каждой зоны:

```text
отдельное виртуальное устройство;
свои датчики;
свои исполнительные каналы;
свой контекст в логе.
```

Пример контекста:

```text
[отопление][610_thermostats][детская справа]
```

## 17. Sprut.hub и логика WB

Для критичной логики отопления:

```text
WB = логика, защита, исполнительные команды, журнал;
Sprut.hub = интерфейс, уставки, режимы, отображение.
```

Не переносить отопительную логику в блочные сценарии Sprut.hub, если датчики и исполнительные устройства физически заведены в WB.

## 18. Проверенные практические правила

- Не использовать `isDefined()` в `wb-rules`, если функция не подтверждена на конкретном контроллере.
- Для стабильного интерфейса задавать `order` у важных полей виртуального устройства.
- Один физический канал должен иметь одного писателя.
- При большом изменении скрипта отправлять полный файл целиком.

## 19. Имена виртуальных термостатов и шаблоны Sprut.hub

Для WB-термостатов, которые должны автоматически распознаваться шаблонами Sprut.hub, использовать типовые имена виртуальных устройств:

```text
NL_simple_thermostat_607
NL_combo_thermostat_607
NL_climate_thermostat_607
```

Маски распознавания Sprut.hub:

```text
NL_simple_thermostat_[0-9]{3}
NL_combo_thermostat_[0-9]{3}
NL_climate_thermostat_[0-9]{3}
```

Номер в имени выбирается по проектному номеру. Если один термостат управляет несколькими исполнительными каналами, в имени используется один выбранный номер, а реальные каналы указываются внутри WB-скрипта и в системном журнале.

Смысл типов:

```text
NL_simple_thermostat
1 датчик температуры, 1 уставка, 1 исполнительная команда нагрева.

NL_combo_thermostat
датчик воздуха + датчик пола, уставка воздуха, цель пола при поддержании, цель пола при нагреве, отопление только через пол.

NL_climate_thermostat
датчик воздуха + датчик пола, отопление и охлаждение. Радиатор не обязателен.
```

Если для зоны выбран `NL_climate_thermostat`, в данных зоны должны быть явно указаны данные кондиционера или должно быть явно написано, что охлаждение в текущем этапе не реализуется.
