## 1. Назначение репозитория

Этот репозиторий это рабочая база NEIROLINKS для инженерной интеграции систем автоматизации.

Главная задача репозитория:
- связать физический объект, проект в DWG, кабельный журнал, каналы Wiren Board, MQTT, скрипты, документацию и опыт эксплуатации;
- ускорять настройку новых объектов;
- снижать ручную работу;
- не переносить ошибки между объектами.

Метод работы называется EIM — Engineering Integration Method.

## 2. Формат ответов

Отвечай на русском языке.

Форматируй ответы инженерно:
- кратко;
- структурированно;
- без маркетинга;
- без украшательства;
- без имитации уверенности.

Используй маркировку:
- `[Проверено]` — факт подтверждён прочитанным файлом, командой, логом, diff или сообщением пользователя;
- `[Предположение]` — логический вывод, но не подтверждённый факт;
- `[Не могу проверить]` — данных нет, файл не прочитан, команда не выполнялась, ссылка не открыта.

Если файл не был прочитан, прямо пиши: `файл не прочитан`.
Если ссылка не была открыта, прямо пиши: `ссылку не открывал`.
Если данных нет, не заменяй проверку догадкой.

## 3. Приоритет источников

При конфликте данных используй такой порядок:

1. Фактически прочитанные файлы текущего объекта.
2. Фактически прочитанные конфиги, бэкапы, логи и MQTT-данные текущего объекта.
3. Явное сообщение пользователя в текущей задаче.
4. Документы EIM и общие стандарты репозитория.
5. Опыт других объектов — только как идея, не как факт.

Запрещено переносить между объектами по аналогии:
- MQTT-каналы;
- Modbus-адреса;
- slave_id;
- device id;
- уставки;
- номера входов и выходов;
- датчики;
- назначение реле;
- названия помещений.

## 4. Структура репозитория

Ожидаемая структура:

```text
EIM/                         метод, пайплайн, правила модели
objects/                     реальные объекты
objects/<object>/README.md   описание объекта
objects/<object>/etc/        конфиги и структура контроллера
objects/<object>/etc/wb-rules/
objects/<object>/etc/wb-rules-modules/
objects/<object>/Backup*     бэкапы, архивы, исходные данные
README.md                    общее описание репозитория
```

Объект — это отдельная зона фактов.

Если работа идёт с `objects/05_16_Iset`, не используй факты из `objects/05_31_Ivolga_13` без явной пометки:
`[Предположение] Взято как опыт другого объекта, применимость не подтверждена`.

## 5. EIM-логика

Основная цепочка EIM:

```text
DWG
↓
Python parser
↓
модель каналов
↓
паспорт объекта
↓
шаблоны скриптов
↓
скрипты конкретного объекта
↓
Wiren Board runtime
```

Ключевой идентификатор канала:

```text
MODULE.CHANNEL
```

Примеры:
- `A06.K1`
- `A06.in1`
- `A05.K2`

Этот идентификатор должен связывать:
- проект;
- кабельный журнал;
- MQTT;
- скрипты;
- документацию;
- исполнительные файлы.

## 6. Правила работы с каналами

Не придумывай каналы.

Канал можно считать подтверждённым только если он найден в одном из источников:
- файл объекта;
- конфиг объекта;
- схема;
- кабельный журнал;
- MQTT-вывод;
- лог;
- прямое сообщение пользователя.

Принцип:
- один канал = один писатель;
- один исполнительный выход не должен управляться несколькими независимыми сценариями;
- если канал пишет Sprut.hub, wb-rules не должен писать туда без явно описанной архитектуры;
- если канал пишет wb-rules, другой скрипт не должен параллельно писать туда же.

При генерации кода всегда отдельно указывай:
- какие каналы читаются;
- какие каналы пишутся;
- где риск конфликта писателей.

## 7. Wiren Board / wb-rules

Код для `wb-rules` писать с учётом реального окружения Wiren Board, а не обычного Node.js.

Использовать стиль ECMAScript 5:
- `var`;
- обычные `function (...) { ... }`;
- простые циклы `for`;
- без стрелочных функций;
- без `let` и `const`, если совместимость не подтверждена.

Запрещено:
- использовать `sleep()`;
- строить бесконечные блокирующие циклы;
- полагаться на глобальные переменные из другого файла `/etc/wb-rules`;
- писать код, который требует современных возможностей JavaScript без проверки;
- создавать скрытые зависимости между сценариями.

Общие функции выносить в `/etc/wb-rules-modules`.
Сценарии объекта хранить в `/etc/wb-rules`.

При изменении скрипта всегда выдавай полный файл целиком, а не фрагмент и не patch.

## 8. Работа с файлами и изменениями

Перед изменением файлов:
1. Прочитай релевантные файлы.
2. Кратко перечисли, что прочитано.
3. Сформулируй план.
4. Укажи, какие файлы будут изменены.
5. Жди подтверждения, если задача не содержит явного указания `правь сразу`.

Не меняй файлы вне целевого объекта без прямого запроса.

Не удаляй и не переписывай бэкапы, архивы, старые конфиги и исходники без прямого разрешения.

Запрещены опасные команды без отдельного подтверждения пользователя:
- `rm -rf`;
- `git reset --hard`;
- `git clean -fd`;
- `git push --force`;
- массовые `chmod -R`;
- массовые `chown -R`;
- удаление `.git`;
- удаление папок `Backup`, `Backup_conf`, `JSON`, `Doc`, `etc`.

## 9. Git и GitHub Desktop

Пользователь работает через GitHub Desktop, который синхронизирует локальные файлы и GitHub.

Не считай, что локальное изменение уже отправлено в GitHub.

После правок всегда дай:
- список изменённых файлов;
- что изменено;
- зачем изменено;
- риски;
- как проверить;
- короткий commit message на русском.

Коммиты, push и pull request выполнять только по прямому запросу пользователя.

## 10. Документация

Документация должна быть:
- инженерной;
- короткой;
- полной по сути;
- пригодной для команды;
- пригодной для повторного использования;
- без рекламных формулировок.

Не использовать стиль “документ о документе”.
Не писать лишнюю теорию.
Не использовать местоимение “мы” в формальных документах, если это не требуется контекстом.

Для объекта разделяй:
- факты объекта;
- неподтверждённые данные;
- решения;
- открытые вопросы;
- задачи на проверку.

## 11. Работа с объектами

Для каждого объекта сначала определить:
- код объекта;
- назначение объекта;
- состав оборудования;
- структуру папок;
- подтверждённые каналы;
- активные скрипты;
- бэкапы и исходники;
- что уже работает;
- что требует проверки.

Если пользователь просит перенести решение с другого объекта:
1. Найди источник решения.
2. Выдели переносимую идею.
3. Отдели объектные факты от общей логики.
4. Запрети автоматический перенос каналов.
5. Адаптируй только после проверки файлов текущего объекта.

## 12. Режимы работ

Для стадии ПНР:
- допустимы эксперименты;
- обязательно указывать риск;
- желательно сохранять старую версию файла.

Для объекта в эксплуатации:
- эксперименты запрещены;
- только безопасные и проверяемые изменения;
- сначала план и diff;
- потом правка;
- потом инструкция проверки и отката.

Если статус объекта неизвестен, считать его объектом в эксплуатации.

## 13. Python и инструменты

Python допустим для:
- парсинга DWG;
- генерации паспортов;
- проверки структуры;
- анализа CSV;
- генерации Markdown/Excel;
- валидации каналов;
- подготовки отчётов.

Скрипты должны быть прозрачными:
- понятные входы;
- понятные выходы;
- без скрытой магии;
- с проверками ошибок;
- с сухими комментариями.

## 14. Формат ответа после работы

После анализа или изменения выдавай:

```text
[Проверено]
Что реально прочитано / выполнено.

[Предположение]
Что является выводом, а не фактом.

[Не могу проверить]
Что не удалось проверить.

[Изменено]
Список файлов и суть изменений.

[Проверка]
Как проверить результат.

[Риски]
Что может пойти не так.

[Commit message]
Короткое сообщение для GitHub Desktop.
```

Если изменений не было, раздел `[Изменено]` заменить на:
`[Изменено] Файлы не менялись`.

## 15. Главный принцип

Работа ведётся не как набор отдельных задач, а как система накопления, стандартизации и применения инженерного опыта между объектами.

Лучший результат — не просто рабочий файл, а решение, которое:
- не ломает текущий объект;
- понятно по источникам данных;
- можно проверить;
- можно откатить;
- можно переиспользовать на следующем объекте без переноса чужих каналов.
